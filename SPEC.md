# RadioCast — Specification

Lightweight, self-hostable 24/7 radio streaming server. Think "OBS for internet radio": it loops a folder of music (VLC-style), composites it with a looping video/GIF background, and pushes a continuous stream to any RTMP ingest (Twitch, YouTube, Kick, custom). Built to run on weak servers.

## 1. Goals

| Goal | How |
|---|---|
| Very low resource usage | A single FFmpeg process does ALL media work. The web panel is a thin controller that does zero media processing. |
| 24/7 uptime | Watchdog auto-restarts FFmpeg on crash with exponential backoff. |
| VLC-like playlist | Drop audio files in a folder (or upload via UI), loop forever, shuffle, reorder. |
| Stream preview | JPEG snapshot every 3 seconds written by the same FFmpeg process (near-zero extra CPU). No transcoded preview stream. |
| Quality control | 480p / 720p / 1080p presets, FPS, video/audio bitrate, x264 speed preset, "CPU saver" mode. |
| Alerts | Stores your alerts overlay URL (StreamElements, Streamlabs, etc.) in settings. |

## 2. Features

### 2.1 Music playlist
- Scans `media/music/` for `.mp3 .flac .ogg .wav .m4a .aac`.
- Upload through the web UI (files land in the same folder).
- Metadata (title, artist, duration) parsed in pure JS (`music-metadata`) — no ffprobe spawns.
- Loop-forever playback via FFmpeg concat demuxer; shuffle toggle; reorder and remove tracks.
- Playlist edits while live show a "Restart to apply" badge — the stream is never interrupted implicitly.

### 2.2 Video background
- Upload a video (`.mp4 .mkv .webm .mov`) or GIF to `media/video/`; select the active one in Settings.
- FFmpeg loops it infinitely with `-stream_loop -1`.
- Fallback when no video is selected: solid dark background generated by FFmpeg (`color` source) — costs almost nothing.

### 2.3 Output (RTMP)
- RTMP server URL + stream key (key is masked in the UI).
- Start / Stop / Restart controls.
- **Test mode**: streams to a null output so you can validate the whole pipeline (and see the preview) without a real stream key.

### 2.4 Quality presets
| Preset | Resolution | Suggested bitrate |
|---|---|---|
| 480p | 854x480 | 1200 kbps |
| 720p | 1280x720 | 2500 kbps |
| 1080p | 1920x1080 | 4500 kbps |

- FPS: 24 / 30 / 60 (default 30 — use 24 on very weak CPUs).
- Audio bitrate: 128 / 192 / 320 kbps AAC.
- x264 preset: `ultrafast` → `medium`. **CPU saver** = 720p / 24fps / ultrafast.

### 2.5 Preview & status
- Dashboard shows a snapshot preview (refreshed every 3s), LIVE badge, uptime.
- Now Playing / Next track computed from track durations + elapsed time (no probing while live).
- FFmpeg stats (bitrate, speed, dropped frames) parsed from `-progress` output.
- FFmpeg log tail (last 200 lines) in a collapsible console.
- All live data delivered over native Server-Sent Events — no websocket library.

### 2.6 Alerts URL
Rendering an HTML alerts page into video requires a headless browser (300+ MB RAM), which contradicts the low-resource goal. RadioCast therefore **stores** your alerts URL — plus a recommended overlay **width/height**, resizable by dragging a preview box in Settings or typing exact pixels — and shows both on the dashboard for use in your platform's own overlay/extension system (e.g. as an OBS Browser Source). Headless-browser compositing is a documented future option.

## 8. Reliability notes (v1.1)
Fixes for track-switch delay, loop-boundary interruptions, and silent stream stalls reported in production use. All of the below were verified against real FFmpeg builds (not just reasoned about), including a 20+ lap soak test of the self-referencing loop:

- **Loop-boundary crash (the main bug)**: the self-referencing playlist file used to be named `.radiocast-playlist.txt`. FFmpeg's concat demuxer rejects any referenced file whose name starts with `.` as `"Unsafe file name"` — **even with `-safe 0`** — so every time playback wrapped around, the audio input died with a demuxing error while the video kept going, which is exactly the "loop causes an interruption / stream starts buffering" symptom. Renamed to `radiocast-playlist.txt` (no leading dot) and the loop is now clean. (We also tried replacing the self-reference trick with FFmpeg's `-stream_loop -1`, matching the background video — testing showed that actually breaks the concat demuxer with `Operation not permitted` on every lap, so the self-referencing pattern is intentionally kept for this input.)
- **Smooth track transitions**: added `-fflags +genpts+igndts` plus an `aresample=44100:async=1,asetpts=N/SR/TB` audio filter so inconsistent sample rates/channel layouts/VBR headers between files in your library can no longer cause an audible stall or click when FFmpeg switches tracks.
- **Stall watchdog**: FFmpeg can occasionally hang (e.g. a wedged RTMP socket) without actually exiting, which the old crash-only watchdog couldn't detect — viewers would see endless buffering with no auto-recovery. The engine now tracks the last `-progress` update and force-restarts FFmpeg if it goes quiet for 25s while the stream should be live.
- **Restart race fix**: `Restart` used to kill the old FFmpeg process and spawn a new one after a fixed 500ms guess, occasionally overlapping two processes fighting over the same RTMP key/playlist/preview file. It now waits for the old process to actually exit before starting the new one.
- **Scheduled maintenance restart**: the self-referencing loop trick, while now crash-free, still leaks roughly one file descriptor per lap (measured empirically). A race-free restart every 6 hours keeps this well away from the OS open-files limit on a real 24/7 stream, at the cost of a sub-second reconnect blip.
- **RTMP backpressure resilience**: `-thread_queue_size` on both inputs and `-max_muxing_queue_size` on the output absorb brief disk/network hiccups on a weak server instead of aborting the whole stream.

## 3. Low-resource design decisions
- **One FFmpeg process** handles video loop + audio concat + encode + RTMP push + preview snapshots (via `tee`/multiple outputs).
- **No database** — all state in `data/state.json`.
- **No websockets** — native SSE.
- **No client-side polling storms** — one SSE connection per open dashboard.
- FFmpeg binary from `ffmpeg-static` (override with `FFMPEG_PATH` env to use system ffmpeg and save 80 MB disk).
- Expected footprint on a 1-vCPU VPS: FFmpeg ~30-60% CPU at 720p ultrafast, panel ~80 MB RAM idle.

## 4. Architecture

```
app/
  page.tsx                Dashboard (preview, status, start/stop, now playing, logs)
  playlist/page.tsx       Playlist manager
  settings/page.tsx       Output, quality, background, alerts URL
  api/stream/route.ts     POST { action: start|stop|restart }
  api/status/route.ts     GET (SSE) status + now playing + log tail
  api/playlist/route.ts   GET list / POST reorder+shuffle / DELETE track
  api/upload/route.ts     POST multipart upload (music | video)
  api/settings/route.ts   GET / PUT settings
  api/preview/route.ts    GET preview.jpg (no-cache)
lib/
  state.ts                JSON state persistence
  scan.ts                 Media folder scanning + metadata
  ffmpeg-args.ts          Builds the FFmpeg command from settings
  streamer.ts             Singleton process manager + watchdog + now-playing clock
media/music/  media/video/  data/    (runtime, gitignored)
```

## 5. Security note
v1 has no built-in auth — run it on a private server or behind reverse-proxy auth (Basic Auth on nginx/Caddy). The stream key is stored in `data/state.json`; keep that directory private.

## 6. Deployment
```bash
pnpm install && pnpm build && pnpm start   # or: node server via pm2/systemd
```
Requirements: Node 20+, Linux. `pnpm install` (via its `postinstall` hook) now runs
`scripts/ensure-ffmpeg.js`, which installs a system ffmpeg (`apt`/`dnf`/`yum`/`apk`/`pacman`,
whichever is available) and writes `FFMPEG_PATH` into `.env` automatically. This matters: the
bundled `ffmpeg-static` binary is a fully static glibc build that is known to segfault instantly
the moment it tries to resolve the RTMP host's DNS on some Linux hosts -- a 100% reproducible
crash-loop on "Start Stream" that has nothing to do with your playlist/settings. System ffmpeg
doesn't have that problem. If the auto-install can't run (no root/sudo, unsupported distro), it
prints instructions instead of failing the install -- re-run it any time with `npm run setup:ffmpeg`,
or set `FFMPEG_PATH` by hand.

## 7. Out of scope (v1)
Headless-browser alert compositing, multi-bitrate simulcast, user accounts, HLS viewer output, audio-only Icecast output.
